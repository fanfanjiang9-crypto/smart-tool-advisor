# 场景化评测框架

## 五维评分体系

### 1. 场景匹配度（权重 30%）

这个工具是否直接解决使用者描述的核心痛点？

**客观层面**：
- 工具的核心功能是什么？
- 它的设计初衷是服务什么人群？

**场景化追问**：
- 使用者的核心痛点，这个工具能解决百分之多少？
- 有没有使用者需要但工具不提供的功能？
- 工具的工作流和使用者的工作流能不能对得上？

**打分标准**：
- 5/5：精准命中核心痛点，几乎是量身定做
- 3/5：部分相关，但需要绕路或用额外工具补漏
- 1/5：名头相关，实际用起来跟使用者的场景对不上

### 2. 上手难度（权重 20%）

以使用者的技术水平，多久能把这个工具跑通、跑顺？

**客观层面**：
- 有没有图形界面？还是纯命令行？
- 文档/教程质量如何？有没有中文资源？
- 从安装到产出第一个有效结果，典型路径多长？

**场景化追问**：
- 使用者的技术水平够不够？（如果使用者是内容创作者，命令行工具天然扣分）
- 使用者有没有时间学？（如果使用者说了"省时间"是核心需求，学习成本高的工具直接警告）
- 有没有更简单但效果接近的替代品？

**打分标准**：
- 5/5：15 分钟内出成果，不需要看文档
- 3/5：需要 1-2 小时学习，但有清晰教程
- 1/5：需要专业背景知识，教程稀少或只有英文

### 3. 性价比（权重 20%）

在使用者的预算范围内，是不是最优解？

**客观层面**：
- 免费版能不能长期用？还是试用陷阱？
- 付费版多少钱？和竞品比如何？
- 有没有隐藏成本（API 超额费、席位费、存储费）？

**场景化追问**：
- 使用者的使用频率（天天用还是偶尔用）决定了值不值
- 使用者的预算范围能不能覆盖
- 如果能省下使用者的时间，省下的时间值多少钱？

**打分标准**：
- 5/5：免费版足够用，或在预算内且 ROI 明显
- 3/5：略超预算，但功能明显强于低价竞品
- 1/5：严重超预算，或性价比不如竞品

### 4. 扩展性（权重 15%）

这个工具用熟了之后，能不能解决更多问题？

**客观层面**：
- 有没有 API？能不能接入自动化流程？
- 有没有插件/生态？
- 能不能团队协作？

**场景化追问**：
- 使用者的工作流会不会扩展？（如果只是解决一个单点问题，扩展性没那么重要）
- 使用者是一个人用还是团队用？
- 使用者有没有提到未来规划？

**打分标准**：
- 5/5：能从单点工具成长为核心工作流平台
- 3/5：本职做得好，但边界清晰，做不了别的
- 1/5：用完就扔的一次性工具

### 5. 可持续性（权重 15%）

这个工具 6 个月后还能不能用？值不值得投入时间学？

**客观层面**：
- 团队背景（创始人/融资/社区）
- 更新频率
- 有没有大厂竞品虎视眈眈

**场景化追问**：
- 使用者的工作对这个工具的依赖度有多高？
- 如果工具突然停服，迁移成本多大？
- 使用者的场景是否长期稳定？（如果是一个临时项目，可持续性不重要）
- **如果使用者有开源偏好**：这个工具的开源协议是什么？能不能自部署？社区贡献活跃吗？

**打分标准**：
- 5/5：大厂出品或活跃开源社区（GitHub 1k+ stars），5 年内不会死
- 3/5：小团队但势头好，至少 1-2 年稳定；或开源但社区较小
- 1/5：个人闭源项目，随时可能停更

### 额外标签：开源/闭源

每个工具评测时，必须在报告中明确标注：

| 标签 | 含义 | 使用者偏好开源时的态度 |
|------|------|----------------------|
| 🟢 开源 | GitHub 公开仓库，可自部署 | 优选 |
| 🟡 半开源 | 部分开源（核心闭源，SDK/客户端开源） | 可用 |
| 🔴 闭源 | 完全闭源，依赖厂商 | 谨慎 |
| ⚪ 免费但闭源 | 不收费但代码不公开 | 标注风险 |

## 综合评分

总分 = Σ(维度得分 × 权重)，满分 5 分。

| 分数 | 含义 |
|------|------|
| 4.0-5.0 | 🏃 强烈推荐，你的场景用它就对了 |
| 3.0-3.9 | 👍 可以用，但注意 XX 短板 |
| 2.0-2.9 | 👀 不太匹配，建议看看替代品 |
| 1.0-1.9 | ❌ 不适合你，别浪费时间 |
